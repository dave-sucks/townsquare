# Social Import + Place Knowledge Engine (Replit Build Spec)

Owner: Dave  
Target: Replit (single backend + Prisma/Postgres)  
Goal: Paste an Instagram Profile URL (and later TikTok), import last N posts, resolve places via Google, create Users/Places/Reviews/Photos, then run an AI tagging + aggregation engine that powers discovery chips like **Classic / Dive Bar / Affordable** and lets users search for **smashburger / birria / rooftop / takeout / delivery** etc.

---

## 0) High-level outcomes

### What “done” looks like
1. **Admin can import a creator by profile URL** (e.g. Instagram profile):
   - Pull last 100 posts
   - Create/Upsert the creator `User`
   - For posts that can be place-resolved:
     - Upsert `Place` using `googlePlaceId`
     - Create `Review` (source = instagram)
     - Create `Photo` rows (carousel supported)
   - For posts that can’t be place-resolved:
     - Keep them as **Unresolved** (review not created yet)
2. **AI tagging engine runs automatically**
   - Extracts review-level signals/tags
   - Aggregates to place-level tags with confidence
   - Updates place “chips” each time a new review is added
3. **Discovery-ready data**
   - Place detail page can show chips grouped by category (Style / Vibe / Price / Food / Features / Occasion)
   - Search can filter by tags and rank by match quality

## 2) New/updated models (Prisma)

### 2.1 Add a generalized import job model (recommended)
This avoids mixing “post-level status” into a single row and supports retries, progress tracking, and manual resolution.

#### `ImportJob`
Tracks a batch import such as “instagram profile last 100 posts”.

Fields:
- `id` (uuid)
- `platform` enum: `instagram | tiktok`
- `type` enum: `profile | post_url | place_seed`
- `input` string (profile URL, etc.)
- `maxPosts` int
- `status` enum: `pending | running | completed | failed | cancelled`
- `createdAt`, `startedAt`, `completedAt`
- Counters: `postsFetched`, `postsProcessed`, `reviewsCreated`, `postsUnresolved`, `postsFailed`
- `error` text nullable

Indexes:
- `platform`, `status`, `createdAt`

#### `IngestedPost`
One row per scraped post. This is the “raw truth” / audit log and enables retries.

Fields:
- `id` (uuid)
- `platform` enum: `instagram | tiktok`
- `importJobId` FK → ImportJob
- `canonicalPostId` string (IG shortcode/postId; TikTok id) UNIQUE per platform
- `url` string
- `authorHandle` string
- `authorPlatformId` string nullable
- `caption` text nullable
- `postedAt` datetime nullable
- `likeCount` int nullable
- `media` jsonb (array of media items: urls, types, dimensions)
- `rawPayload` jsonb (full Apify payload)
- Place resolution fields:
  - `resolvedGooglePlaceId` string nullable
  - `resolveMethod` enum: `geotag | caption | hashtag | ai | manual | none`
  - `resolveConfidence` float nullable
  - `resolveCandidates` jsonb nullable (top 3 Google candidates)
- Downstream link:
  - `reviewId` FK → Review nullable
- `status` enum: `new | processed | unresolved | failed`
- `error` text nullable
- `createdAt`, `updatedAt`

Unique constraints:
- `(platform, canonicalPostId)` unique

Indexes:
- `status`, `authorHandle`, `resolvedGooglePlaceId`

> If you want to keep `InstagramImport`, you can:
> - either replace it with these, or
> - make `InstagramImport` act as `ImportJob` and store per-post stuff in `IngestedPost`.

### 2.2 Add review-level tag evidence + place-level aggregates

#### Option A (cleaner): add `ReviewTag` + `PlaceTagAggregate`
Keep your existing `PlaceTag` for user/manual tags if you want, but use these for AI-driven truth.

##### `ReviewTag`
Evidence extracted from a single review/post.

Fields:
- `id` uuid
- `reviewId` FK → Review
- `tagId` FK → Tag
- `source` enum: `ai | user | manual`
- `confidence` float (0..1)
- `createdAt`

Unique:
- `(reviewId, tagId, source)` unique (or just `(reviewId, tagId)` if you don’t need multi-source)

Index:
- `reviewId`, `tagId`

##### `PlaceTagAggregate`
Aggregated truth shown in UI.

Fields:
- `id` uuid
- `placeId` FK → Place
- `tagId` FK → Tag
- `confidence` float (0..1)
- `evidenceCount` int
- `evidenceScore` float
- `lastComputedAt` datetime
- `isSuppressed` boolean default false (manual override to hide)
- `sourceBreakdown` jsonb nullable (optional: counts by source)

Unique:
- `(placeId, tagId)` unique

Indexes:
- `placeId`, `confidence`

> If you prefer minimal changes: you can store aggregated tags in existing `PlaceTag` and add `evidenceCount/evidenceScore` columns.
> But long-term, separating evidence (ReviewTag) from truth (PlaceTagAggregate) is worth it.

## 4) System architecture in Replit

### 4.1 Two responsibilities
1. **HTTP API** (Next/Express/whatever you have)
2. **Worker loop** (runs in same codebase; processes queued jobs in DB)

### 4.2 Job queue approach (DB-backed)
Create a simple `Job` table:

`Job`
- id uuid
- type enum:  
  - `IMPORT_PROFILE`  
  - `PROCESS_POST`  
  - `ENRICH_REVIEW`  
  - `UPDATE_PLACE_AGGREGATES`  
  - `REFRESH_PLACE_SUMMARY`
- payload jsonb
- status enum: `queued | running | completed | failed`
- attempts int
- runAfter datetime (for backoff)
- error text
- createdAt, updatedAt

Worker:
- every X seconds:
  - claim next `queued` job with `runAfter <= now()`
  - set status `running`
  - execute
  - mark completed/failed, increment attempts

This avoids needing Redis/queues on day 1.
