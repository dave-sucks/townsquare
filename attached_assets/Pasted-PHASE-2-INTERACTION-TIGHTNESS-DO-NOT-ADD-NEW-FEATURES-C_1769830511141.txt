PHASE 2 — INTERACTION TIGHTNESS (DO NOT ADD NEW FEATURES)

Context:
We already have a working web-first Beli clone with:
- Google Places search
- Saving places (Want / Been)
- A split screen with a list on the left and a Google Map on the right
- Pins rendering correctly from saved places
- Data persisted in places + saved_places tables

This phase is NOT about design polish, recommendations, social feed, or new features.
This phase is ONLY about tightening the interaction between:
LIST ↔ MAP ↔ PLACE SELECTION

HARD UI RULES (UNCHANGED):
1) All non-map UI must use shadcn/ui components only.
2) Do NOT add ad-hoc Tailwind utility classes throughout pages.
   - If layout or styling changes are needed, implement them via small wrapper components using cva variants (e.g. AppShell, MapShell, PlaceRow).
3) The Google Map canvas itself is allowed to be a plain div (map libraries are exempt), but all overlays, panels, cards, tabs, and controls must be shadcn.
4) Do not introduce new UI libraries.

SCOPE — WHAT TO BUILD IN THIS PHASE:

A) Single source of truth for selection
- Introduce a single piece of state: selectedPlaceId.
- This state must control:
  - Which list row is active
  - Which map marker is active
  - What the map is centered on
  - What place preview (if any) is visible
- There should be no duplicated “selected” state across components.

B) List → Map coupling
- Clicking a place in the left list:
  - Sets selectedPlaceId
  - Pans the map to that place’s lat/lng
  - Sets a reasonable zoom level (street / neighborhood, not world view)
  - Visually highlights the selected list row
- The list should behave as a controller for the map.

C) Map → List coupling
- Clicking a map marker:
  - Sets selectedPlaceId
  - Highlights the corresponding list row
  - Scrolls the list to bring that row into view
- Marker interaction should feel symmetrical with list interaction.

D) Marker states
- Markers must visually encode at least three states:
  1) WANT
  2) BEEN
  3) SELECTED (overrides the above)
- This can be implemented via:
  - different marker colors
  - different marker icons
  - size or z-index changes
- Do NOT overdesign markers; just make states clearly distinguishable.

E) Lightweight place preview (map overlay)
- When a place is selected (from list or map), show a lightweight preview:
  - Implemented as a shadcn Card overlaid on the map
  - Shows:
    - place name
    - address
    - current status (Want / Been)
    - quick actions (Mark Been / Remove / Open details)
- This is NOT a full modal and NOT a route change.
- Clicking “Open details” can still navigate to the full place page.

F) Layout tightening (function-first, not visual polish)
- Ensure the app behaves like a split-view product:
  - Header remains visible
  - Left list scrolls independently (use ScrollArea)
  - Map fills the remaining vertical space
- Avoid large empty whitespace below the map or list.
- Do not redesign typography or colors yet.

WHAT NOT TO DO IN THIS PHASE:
- Do NOT add recommendations
- Do NOT add social feed or follows
- Do NOT add lists-of-lists
- Do NOT redesign the visual theme
- Do NOT refactor the database unless required for selection state
- Do NOT add filters or sorting beyond what already exists

ACCEPTANCE CRITERIA (ALL MUST BE TRUE):
- Clicking a list item pans and highlights the map marker
- Clicking a marker highlights and scrolls the list
- Want vs Been markers are visually distinguishable
- Selected place is clearly indicated in both list and map
- A small place preview card appears on selection
- No new Tailwind class soup introduced in pages

WORKFLOW:
1) Implement selectedPlaceId and wire it through list and map
2) Tighten bidirectional interactions
3) Add marker state differentiation
4) Add map overlay preview card
5) Only then adjust layout spacing as needed

After finishing, summarize:
- Where selectedPlaceId lives
- How list ↔ map synchronization works
- Any new wrapper components introduced

Do not proceed beyond Phase 2 scope.
