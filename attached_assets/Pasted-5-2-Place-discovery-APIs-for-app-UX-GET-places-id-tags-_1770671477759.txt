7) Place resolution spec (Google matching)
7.1 Inputs available per post

From Apify payload, try to capture:

Instagram location tag name (if present)

location address fields (if present)

caption text

hashtags

profile city (optional)

7.2 Resolution ladder

Try in this order:

Geotag (best)

If post includes a location tag:

Google Places Text Search using location name + city context (NYC default)

choose best candidate by:

name similarity

distance if lat/lng from post exists

address match

if confident: resolveMethod=geotag

Caption name extraction

Use a lightweight AI extraction prompt:

“Return the most likely venue name and neighborhood/city from this caption”

Then run Google Places Text Search on that extracted venue string

Hashtag extraction

If hashtags look like venue names (#rubirosa):

try search

Fallback AI

Ask AI: “What place is this likely referring to?”

Then search

If still low confidence:

unresolved

7.3 Confidence thresholds

>= 0.85 => auto-create Review

0.60 - 0.85 => unresolved with candidates (manual resolve)

< 0.60 => unresolved, no candidates or weak candidates

8) AI Tagging Engine
8.1 Taxonomy rules

You MUST seed a controlled vocabulary in TagCategory + Tag.

Categories suggested:

Style (classic, trendy, modern, old_school)

Vibe (dive_bar, classy, cocktail_bar, lively, quiet, romantic, group_friendly)

Price (cheap, affordable, pricey, expensive)

Food Type (smashburger, burger, birria_tacos, tacos, pizza, pasta, ramen, sushi, bbq, wings, etc.)

Important: Use slug as the canonical ID in prompts.

8.2 Review-level extraction: ENRICH_REVIEW(reviewId)

Inputs:

review.note (if manual)

review.socialPostCaption (if imported)

place name + neighborhood (for context)

optional: google types

Output:

JSON containing category → tag slugs with confidences

optional: “new tag suggestions” list (not auto-created)

Write:

Insert ReviewTag rows for each predicted tag with confidence

Keep only tags with confidence >= 0.55 at review level (tunable)

8.3 Place aggregation: UPDATE_PLACE_AGGREGATES(placeId)

Deterministic computation (no AI needed):

Load all ReviewTag for reviews of that place

Group by tagId

Compute:

evidenceCount = number of distinct reviews contributing

evidenceScore = sum(confidence * sourceWeight)

Convert to final place confidence:

simplest: placeConfidence = min(1, evidenceScore / K) where K is scaling factor

or use sigmoid if desired

Write to PlaceTagAggregate upserting by (placeId, tagId)

Recommended source weights:

manual: 1.2

user: 1.0

ai: 0.8

google: 0.6

Promotion/display rules:

Only show tag if:

confidence >= 0.70 OR evidenceCount >= 2

Some tags require higher confidence (e.g. rooftop, delivery) since they must be factual:

rooftop/delivery/takeout: require confidence >= 0.80 OR evidenceCount >= 2

8.4 Place summary refresh: REFRESH_PLACE_SUMMARY(placeId) (optional early, recommended later)

Inputs:

Place name, neighborhood, google types, top 10 tags + confidence

A few representative captions/notes (top 3-5)
Output:

Place.aiSummary (2–3 sentences)

Place.topChips (pick 1–2 from Style, 1–2 from Vibe, 1 from Price)

Run:

only when place has >= 2 reviews and every +5 new reviews

or on demand in admin

12) Implementation notes (gotchas to avoid)

Do not auto-resolve low confidence places

Wrong place mappings poison everything.

Dedup hard

User: match by instagramId or instagramHandle

Place: match by googlePlaceId

Review: match by instagramShortcode (unique per platform) so you don’t import duplicates

Store raw payloads

You’ll need them when an actor changes output format.

Keep tagging vocab constrained

Prevent tag explosion; add “suggested new tags” as a manual review step.

Separation of evidence vs truth

ReviewTag = evidence

PlaceTagAggregate = truth shown in UI

Cost control

Tag extraction per review is cheap

Place summary refresh should be occasional
