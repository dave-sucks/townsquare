Assume Milestone 4 is complete:

- Follow/unfollow works
- Activity rows are written for save/been/add-to-list/list-create
- /places/[id] and /u/[username] exist and are linked everywhere
- Dashboard/map preview still works

Now implement REVIEWS + PHOTOS so the app becomes “Beli-like” beyond saving pins.

HARD UI RULES (UNCHANGED):
1) All non-map UI MUST use shadcn/ui components only.
2) Do NOT add ad-hoc Tailwind class strings throughout pages.
   - If layout/styling changes are needed, implement wrapper components using cva variants.
3) Map canvas can remain a plain div; all overlays/panels/menus/dialogs must be shadcn.
4) No new UI libraries.

SCOPE OVERVIEW:
- Users can create/edit/delete a review for a place (1 review per user per place).
- Reviews include: rating (0–10), optional note, optional visited date, and photos.
- Photos are stored via an object storage approach (pick one that works in Replit).
  Preferred:
   - Upload to Cloudflare R2 OR AWS S3-compatible storage using signed URLs.
  Acceptable for MVP:
   - Upload to a simple server endpoint and store on disk (ONLY if Replit environment supports persistent storage reliably).
- Feed includes REVIEW_CREATED activities.
- Place pages show:
  - your review editor section
  - a reviews list (from you + people you follow first; optionally all users later)
  - a photo gallery

DATABASE (Prisma) — ADD THESE MODELS:
A) Review
- id (uuid)
- userId (fk -> User)
- placeId (fk -> Place)
- rating int (0..10)
- note string (optional)
- visitedAt datetime (optional)
- createdAt
- updatedAt
Constraints:
- unique(userId, placeId)

B) Photo
- id (uuid)
- userId (fk -> User)
- placeId (fk -> Place)
- reviewId (fk -> Review, nullable) // allow photos attached to a review
- url string (required)            // final CDN/object URL
- width int (optional)
- height int (optional)
- createdAt
Indexes:
- placeId, createdAt DESC
- reviewId, createdAt DESC

C) Activity additions
- Ensure Activity.type enum includes REVIEW_CREATED
- When a review is created (or rating/note changes), write Activity row:
  type = REVIEW_CREATED
  actorId = userId
  placeId = placeId
  metadata can include rating snapshot

IDEMPOTENCY RULE (UNCHANGED):
- Do not duplicate REVIEW_CREATED activities due to retries.
- Use the same dedupe approach from Milestone 4 (dedupeKey or transaction check).

UPLOAD ARCHITECTURE:
Implement photo upload in a clean, MVP-friendly way:
Option A (preferred):
- API route: POST /api/uploads/sign
  - returns a signed upload URL + final public URL
- Client uploads file directly to storage using signed URL
- Then client calls server action to create Photo record with the final URL
Option B (acceptable MVP if you must):
- API route: POST /api/uploads
  - accepts multipart form upload
  - stores file
  - returns public URL
- Then server creates Photo record

Pick ONE option and implement end-to-end.
